# FlowForge v0.5 — Token 优化需求文档

> **核心目标**：将多 Agent 协作流程的 Token 消耗降低 60%-80%。  
> **问题根源**：当前 Agent 之间通过"值传递"（完整文件内容）协作，导致上下文膨胀、成本失控。

---

## 一、现状问题分析

### 1.1 Token 消耗热点

| 热点 | 现象 | 占比 |
|------|------|------|
| 文件内容传递 | Worker A 读文件 → 把几万 Token 文本原样传给 Worker B | ~40% |
| Worker 啰嗦输出 | "好的，我已读取文件，内容如下..." + 完整复述 | ~25% |
| 全量工具定义 | 每个 Worker 都挂着全套工具 Schema（read/write/bash/search） | ~15% |
| 对话历史膨胀 | Supervisor-Worker 多轮对话无限增长 | ~20% |

### 1.2 根因

FlowForge 的节点之间通过 `state["node_outputs"][node_id]["output"]` 传递**完整字符串内容**。当上游节点读了一个 5000 行的代码文件，这 5000 行的文本就被写入 state，下游节点再把它拼进自己的 prompt——Token 消耗呈**乘法效应**。

---

## 二、改进方案

### 🗄️ 第一刀：工件存储（Artifact Store）

**目标**：Agent 之间不再传递文件内容，只传引用。

```
Before（值传递）:
  Worker A → "我读了 config.py，内容如下：[5000行代码全文]"
  Worker B ← 拼入上下文：[5000行代码全文]
  Token 消耗：5000 × 2 = 10000+

After（引用传递）:
  Worker A → "已读取并存储 config.py [REF: artifact://config.py#v1]"
  Worker B ← 收到引用，按需读取 [REF: artifact://config.py#v1#L42-L58]
  Token 消耗：~50 × 2 + 按需读取 = ~200
```

#### 技术方案

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| Redis | 性能最佳，支持 TTL | 依赖外部服务 | ⭐⭐⭐ |
| 本地 SQLite | 零依赖，持久化 | 并发性能一般 | ⭐⭐ |
| **内存 dict + 文件缓存** | **零依赖，开发快** | 进程重启丢失 | ⭐⭐⭐ v0.5 首选 |

**v0.5 推荐方案**：Python 模块级 `dict` 作为 Artifact Store，Key 格式：

```
artifact:{execution_id}:{node_id}:{filename}:{version}
```

#### 数据结构

```python
@dataclass
class Artifact:
    key: str                    # artifact:exec_xxx:node_a:config.py:v1
    file_path: str              # 原始文件绝对路径
    content: str                # 完整内容（只在 Store 里存一份）
    line_count: int
    metadata: dict              # {mtime, size, encoding, checksum}
    created_by: str             # node_id
    created_at: float           # timestamp
    ttl: int = 300              # 5 分钟自动过期
```

#### API 设计

```python
class ArtifactStore:
    def put(self, key: str, file_path: str, content: str, metadata: dict) -> str
    def get(self, key: str) -> Artifact | None
    def get_range(self, key: str, start_line: int, end_line: int) -> str
    def list_keys(self, execution_id: str) -> list[str]
    def cleanup(self, execution_id: str) -> None
```

#### 引用语法

Agent 之间使用统一的引用格式：

```
[REF: artifact://{key}]
[REF: artifact://{key}#L10-L50]
[REF: artifact://{key}#func:handle_login]
```

Adapter 层负责拦截 `[REF: ...]` 语法，自动从 ArtifactStore 取数据注入上下文。

#### 预期效果

- 文件内容传递 Token 消耗：**降低 95%+**
- Worker 上下文不再包含原始文件内容
- 引用语法对 Agent 透明（模型原生理解 `[REF:]` 格式）

### 📜 第二刀：压缩摘要器（Summarizer Proxy）

**目标**：Worker 只返回结构化 JSON，Supervisor 只看干练结果。

```
Before（啰嗦输出）:
  "好的，我已经分析了 config.py 文件。这个文件包含了数据库配置，
   日志配置，以及各种环境变量设置。我注意到第 42 行的 DATABASE_URL
   使用了硬编码的密码，这是一个安全问题。另外第 58 行的 DEBUG=True
   在生产环境中不应该启用...[继续 2000 Token]"

After（结构化输出）:
  {
    "status": "ok",
    "insights": [
      {"severity": "high", "type": "security", "line": 42, "desc": "硬编码密码"},
      {"severity": "medium", "type": "config", "line": 58, "desc": "DEBUG=True"}
    ],
    "files_touched": ["config.py"],
    "summary": "发现 2 个问题：1 个高危安全漏洞，1 个配置建议"
  }
```

#### 技术方案

采用**两层压缩**策略：

**Layer 1 — Prompt 强制约束**（免费）

在 Worker 的 system_prompt 中强制要求 JSON 输出：

```
OUTPUT FORMAT (MANDATORY):
You MUST output ONLY valid JSON. No explanations, no markdown, no code blocks.

{
  "status": "ok" | "error",
  "result": "<concise result, max 200 chars>",
  "insights": [{"type": "...", "desc": "...", "confidence": 0.0-1.0}],
  "refs": ["artifact://..."]
}

DO NOT include any text outside this JSON structure.
VIOLATION of this format will cause your output to be DISCARDED.
```

**Layer 2 — 小模型二次压缩**（付费，按需启用）

当 Worker 输出仍然超过阈值（默认 500 Token）时，用轻量模型做摘要：

```python
class SummarizerProxy:
    def __init__(self, model: str = "deepseek-chat", max_tokens: int = 100):
        self.model = model
        self.max_tokens = max_tokens

    async def compress(self, worker_output: str, task_context: str) -> str:
        """将 Worker 输出压缩到 max_tokens 以内"""
        if len(worker_output) < 500:
            return worker_output  # 不需要压缩
        # 调用小模型做提取式摘要
        ...
```

#### 输出 Schema 按节点角色定制

| 角色 | 强制 JSON Schema |
|------|-----------------|
| Worker | `{"status", "result", "refs"}` |
| Dispatcher | `{"plan": [{"worker_id", "task", "expected_refs"}]}` |
| Aggregator | `{"answer", "sources": [{"ref", "relevance"}]}` |

#### 预期效果

- 单次 Worker → Supervisor 传递 Token：**降低 70%-90%**
- Supervisor 上下文更干净，决策更准确
- 结构化输出可直接用于前端展示

### 🛠️ 第三刀：工具定义按需加载

**目标**：每个 Worker 只拿到它需要的工具，而不是全套。

```
Before:
  Worker A（只读文件）→ 挂着 [read_file, write_file, execute_bash, search, glob, grep, web_fetch, ...]
  System Prompt 中工具定义占 ~800 Token

After:
  Worker A（只读文件）→ 挂着 [read_file, grep]
  System Prompt 中工具定义占 ~150 Token
```

#### 技术方案

在 `PromptGenerator` 中引入**工具分配策略**：

```python
TOOL_ASSIGNMENT = {
    "dispatcher": [],                    # 只分配任务，不需要工具
    "aggregator": [],                    # 只汇总，不需要工具
    "reader_worker": ["read_file", "grep", "glob"],
    "writer_worker": ["read_file", "write_file", "bash"],
    "researcher": ["web_search", "web_fetch"],
    "code_worker": ["read_file", "write_file", "bash", "grep"],
}
```

**分配逻辑**：在 Auto Prompt 生成时，根据任务描述关键词推断 Worker 的角色子类型：

```python
def classify_worker_role(task_description: str) -> str:
    keywords = {
        "reader_worker": ["读取", "查看", "检查", "分析", "review"],
        "writer_worker": ["修改", "写入", "创建", "生成", "重构"],
        "researcher": ["搜索", "查找", "调研", "research"],
        "code_worker": ["实现", "修复", "开发", "debug"],
    }
    ...
```

#### OpenCode API 适配

OpenCode 的 session 创建时指定允许的工具列表：

```json
POST /session
{
    "model": {"providerID": "deepseek", "id": "deepseek-v4-pro"},
    "tools": ["read_file", "grep"],   // 只挂这两个
    "max_steps": 5
}
```

#### 预期效果

- 每个 Worker 的 System Prompt：**减少 500-1000 Token**
- 10 个 Worker 的协作流程：**节省 5000-10000 Token**
- Worker 行为更聚焦（工具少了，不会乱跑）

### 🔄 第四刀：历史消息滑动窗口

**目标**：对话历史只保留最近 N 轮，丢弃中间闲谈。

```
Before（无限增长）:
  Turn 1: Supervisor 分配任务 [500T]
  Turn 2: Worker 确认收到 [200T]
  Turn 3: Worker 读取文件 [800T]
  Turn 4: Worker 报告结果 [600T]
  Turn 5: Supervisor 确认 [200T]
  Turn 6: Supervisor 分配下一个任务 [500T]
  ...共 50 轮，上下文 30000T

After（滑动窗口 K=3）:
  保留：任务目标（首轮，不可丢弃）
  保留：最近 3 轮对话
  丢弃：中间所有"确认收到"、"心路历程"、"中间结果"
  上下文：~3000T
```

#### 技术方案

```python
class SlidingWindow:
    def __init__(self, max_turns: int = 5, preserve_first: bool = True):
        self.max_turns = max_turns
        self.preserve_first = preserve_first  # 始终保留任务目标

    def trim(self, messages: list[dict], task_goal: str) -> list[dict]:
        """裁剪消息列表到滑动窗口大小"""
        if len(messages) <= self.max_turns + 1:
            return messages

        # 保留首条（任务目标）
        head = messages[:1] if self.preserve_first else []

        # 保留最近 K 轮
        tail = messages[-(self.max_turns):]

        # 中间插入摘要标记
        summary = {
            "role": "system",
            "content": f"[{len(messages) - self.max_turns - 1} earlier turns omitted for brevity]"
        }

        return head + [summary] + tail
```

#### 滑动窗口大小建议

| 场景 | 窗口大小 | 理由 |
|------|---------|------|
| 简单文件读取/分析 | 3 轮 | 读取→分析→报告，不需要更多 |
| 代码修改/重构 | 5 轮 | 可能需要多次 read→write→verify |
| 复杂多步任务 | 8 轮 | 需要保留中间决策上下文 |

#### 与 LangGraph 集成

在 `executor.py` 的每个节点调用前执行裁剪：

```python
# 每个节点执行前
trimmed_messages = sliding_window.trim(
    messages=state.get("messages", []),
    task_goal=state.get("task_goal", ""),
    max_turns=get_window_size(node_type, node_config)
)
```

#### 预期效果

- 长流程（10+ 节点）Token 消耗：**降低 40%-60%**
- 短流程（3-4 节点）基本不受影响
- 窗口内信息密度大幅提高

## 四、实施优先级

| 优先级 | 改动 | 理由 |
|--------|------|------|
| **P0** | 结构化 JSON 输出约束 | 零依赖，纯 prompt 改动，立竿见影 |
| **P0** | 工具按需加载 | 改动小（prompt_generator.py + client.py），效果显著 |
| **P1** | 工件存储（内存版） | 需要新增模块，但完全解耦文件传递 |
| **P1** | 滑动窗口 | 改动在 executor 层，风险可控 |
| **P2** | 小模型二次摘要 | 需要额外 API 调用，v0.5 先做 Layer 1 |

## 六、风险与注意事项

1. **JSON 输出不严格**：部分模型可能不遵守 JSON-only 约束 → 加 `json.loads()` 解析兜底 + 重试
2. **引用断裂**：ArtifactStore TTL 过期导致引用失效 → Worker 返回时携带 fallback 摘要
3. **滑动窗口丢失关键信息**：窗口过小导致 Supervisor 遗忘重要上下文 → 窗口大小可配置，task_goal 永久保留
4. **OpenCode tools 白名单 API 兼容性**：需确认 OpenCode 是否支持 per-session tools 过滤 → 如不支持，在 prompt 层面禁止使用非授权工具
